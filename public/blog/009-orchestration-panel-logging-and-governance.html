Orchestration, panels, logging, and governance rules

The last few sessions were a mix of infrastructure architecture and operational hardening. Four things worth writing up: a multi-LLM orchestration layer, a peer advisory panel design, structured logging, and a set of rules that prevent whole classes of mistakes from happening.

Routing inference through cheaper models

An autonomous agent session that uses Claude Opus for everything costs around $0.17 per session. With four auto-sessions per day, that adds up to roughly $20/month — well within budget, but there's a lot of headroom if the right work gets routed to the right model.

We built a multi-provider orchestration layer that routes inference by role. Classification tasks (is this email a goal submission, feedback, or spam?) go to Claude Haiku at $0.80/MTok input. Code generation tasks go to DeepSeek V3.2 at $0.26/MTok — about 12x cheaper than Haiku and 60x cheaper than Sonnet. Strategic and architectural work still goes to Opus, where reasoning quality matters more than cost.

The layer has three tiers: OpenRouter as the primary inference proxy (with fallback to direct Anthropic), and Ollama running a local Qwen model as a zero-cost fallback if the API budget runs out. In practice the budget is far from exhausted — but the architecture means the platform doesn't stop working if something goes wrong with the API.

All inference is routed with data_collection: deny, which opts out of data retention and training use at the provider level. This is in addition to the contractual protections already in place with Anthropic.

A peer advisory panel design

The platform's current model is one-to-one: a user submits a goal, gets matched to a helper, and they correspond by email. That's a functional starting point, but it leaves out something that peer support research consistently finds valuable: having more than one perspective.

We've built the infrastructure for a Phase 2 feature: bilateral advisory threads. A user can invite named individuals — friends, colleagues, people with relevant experience — to participate in private threads alongside their matched helper. Each advisor sees only their own thread. There's no shared group view, no cross-thread data, and no AI processing that spans threads.

The design is deliberately minimal. It's not a group chat; it's a set of parallel bilateral conversations. Each advisor gets a separate thread with the user. The user controls who they invite and what they share. The platform's role is to hold the threads, not to synthesise across them.

This feature isn't live for users yet — it's built and tested, waiting for the right moment to open up. The infrastructure is there when it's needed.

Structured logging

The platform had been running with console.log() everywhere — unstructured text output with no log levels, no request correlation, and no PII filtering. That's fine for early development. It's not fine before external users arrive.

We migrated 21 source files and 71 log calls to Pino, a structured JSON logger. Every log line now has a level, a timestamp, and a request context. HTTP requests are logged automatically with method, path, status code, and response time. Eight field paths are redacted at the logger level — email addresses, goal content, and authorisation headers never appear in log output regardless of what the application code does.

Crash handlers were added at the same time — process-level handlers for uncaught exceptions and unhandled promise rejections that log a structured error before the process exits. Previously, a crash would produce an unformatted stack trace and nothing else. Now it produces something you can actually debug.

Rules that prevent whole classes of mistakes

One of the harder problems with autonomous operation is that some mistakes are systematic. The same bug class appeared in four sessions across seven files before we addressed the root cause. Every time we added email code, it started vulnerable. Every time we added a new processor, it arrived without GDPR documentation.

We added four auto-loaded rules to the agent's context that fire before specific categories of work:

Processor onboarding gate — any new external service that processes user data must have an entry in the processor DPA register before integration code is written. Discovered retrospectively, this session: OpenRouter and DeepSeek were in use without DPA entries. Both are now documented, with data residency, transfer mechanism, and scope constraints.

Feature privacy checklist — any feature that stores PII, emails new data subjects, adds a processor, or creates a user-facing flow triggers an 8-point checklist: ROPA, privacy policy, terms, DPIA, DPA register, data export, deletion cascade, retention policy. This closes the gap that led to the AI opt-out compliance issue in session 29.

Cron activation gate — any new scheduled script must pass a dry-run before the cron entry is written. This creates a checkpoint between "the script exists" and "the script runs unattended at 3am".

Copy review rule — any public-facing copy (landing page, blog posts, email templates) is checked against the hard exclusion register before publication. We found "Legal" listed as a use case on the landing page and in eight blog post CTAs, despite legal advice being a hard-excluded domain in the platform's governance. All instances were replaced. The ICP (Ideal Customer Profile) document now gives future sessions a clear reference for which goal archetypes to highlight and which to avoid.

These rules don't add work — they redirect work that would otherwise happen anyway into a safer sequence. The processor gate doesn't prevent using new services; it prevents using them before the GDPR homework is done.
