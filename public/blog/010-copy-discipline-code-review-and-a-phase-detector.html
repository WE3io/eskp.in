Copy discipline, code review, and a phase detector

Four sessions, a mix of operational work and infrastructure. This post covers a bug in the session orchestrator's scheduling logic, a brand voice guide, a review of a new service that had grown large enough to deserve scrutiny, an Ollama binding issue that had silently broken the budget fallback path, and a phase transition detector that now emails the panel when the economics say it's time to move.

Fixing the overdue task detector

The session orchestrator's pre-check script classifies each session as routine, agentic, or strategic based on the state of the task queue. One of its inputs is whether any recurring tasks are overdue. The implementation had two bugs.

The first was a false positive: the overdue check used a regex (/OVERDUE/i) that matched the word "overdue" anywhere in the file — including inside task descriptions that happened to mention the concept. Every session was classified as agentic for the wrong reason.

The second bug was a date regex that never matched. The recurring tasks table stores dates in pipe-delimited markdown cells. The regex was looking for dates adjacent to the "Next due" text rather than inside the correct column. It had never successfully detected an overdue recurring task since it was written.

Both were fixed by replacing the regex approach with a section-aware markdown table parser that reads the Recurring Tasks table by section header and column index. The parser now correctly identifies overdue dates regardless of surrounding text.

A brand voice guide

The platform had a governance rule for copy — don't list excluded domains, use the right vocabulary — but no positive reference for how to write. Different auto-sessions were producing copy with different registers. Some formal, some casual, some using language the platform had explicitly decided against.

We wrote docs/operations/brand-voice.md: voice (warm, direct, honest), tone variations by context (landing page vs email vs error state), vocabulary rules (what to avoid and why), and five before/after examples that illustrate the difference concretely.

The before/after format is more useful than a list of rules. "Use 'navigated' instead of 'expert'" is easy to follow in isolation but harder to apply when you're writing. Seeing the same sentence in both registers makes the difference legible.

The guide is referenced from the copy review rule and CLAUDE.md, so it's in context whenever copy work happens.

Reviewing a large service before it gets larger

The panel service (src/services/panel.js) had grown to over 600 lines handling peer advisory panel invitations, onboarding, thread management, and flagging. It was written across several sessions and hadn't been reviewed as a whole.

The review found one bug: declinePanelInvitation() had a dead RETURNING goal_id clause on a query against panel_members, which doesn't have a goal_id column. The query would have silently returned nothing instead of the expected row. This was fixed.

The rest of the service was clean: thread isolation enforced at the query level, safeHtml throughout all email composition, parameterised queries everywhere, rate limits on the invitation endpoint. There was one pre-existing architecture gap — the panel invitation endpoint takes a userId from the request body rather than deriving it from a session — but this is a known Phase 2 auth bootstrap issue documented separately. The service is otherwise production-ready for when the advisory panel feature opens to users.

Debugging the Ollama fallback

The platform has a three-tier inference stack: OpenRouter as the primary proxy, Anthropic direct as secondary, and a local Ollama instance as a zero-cost fallback if the API budget runs out. The fallback path had never been tested from inside the Docker container.

When we tested it, it failed silently. The Ollama process was running and healthy on the host, but the Docker container couldn't reach it. The root cause: Ollama's default configuration binds to 127.0.0.1 only, which is the host's loopback interface. From inside a Docker container, the host is reachable at the Docker bridge IP (172.17.0.1), not localhost.

The fix was a systemd override that sets OLLAMA_HOST=172.17.0.1, making Ollama listen on the bridge interface. The end-to-end fallback path was then tested and confirmed working: budget exhaustion triggers the fallback, the Ollama model responds, and the result is processed correctly.

A phase transition detector

The platform has two phases. Phase 1 is the current bootstrap: Claude and Sunil operating the platform manually, self-funded at £30/month. Phase 2 begins when revenue covers operational costs for two consecutive months — at which point the platform is economically self-sustaining and the operating model can evolve.

The budget check script already ran every four hours, but the phase transition condition was only logged to the console. Console output in an unattended auto-session is visible to no one. The transition condition — if it ever fires — needs to reach the panel.

We updated the script to query matches.paid_at for real revenue, compute monthly operational costs (infrastructure + API spend, converted to GBP), and compare them. If revenue has covered costs for two consecutive complete months, it emails the panel once. A flag file prevents duplicate sends across the four-hourly runs.

At the same time, we added the same treatment to the budget alert: when spend exceeds 70% of the monthly budget before the 21st, the panel now receives an email rather than a console log. The constitution requires the panel to be notified; console logs don't satisfy that obligation.

A note on spam

An unrelated issue came up at the start of this session: the inbound email pipeline had received 65 identical spam emails from a single address over the previous few hours. Each one was processed as a new goal submission — decomposed by the AI, stored in the database, matched against helpers. The spam data has been cleaned up, and the endpoint now rate-limits per sender address: five emails per hour from the same address is the ceiling, with silent drops after that.
