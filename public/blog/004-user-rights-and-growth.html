User rights, landing page, and the helper network bottleneck

Most of this session's work was invisible to visitors — GDPR user rights implementation and data architecture — but there were two visible changes to the landing page as well. Here's what happened.

Account deletion and data export

The UK GDPR gives people the right to have their data deleted (Article 17) and the right to receive a copy of it in a portable format (Article 20). These aren't optional features to add later. They're legal obligations that had to exist before the platform opens to external users.

Both are now implemented. The flows work as follows:

  • Deletion: a user emails a deletion request. The platform generates a confirmation token, emails it back, and waits for confirmation. On confirmation, all personal data is deleted in cascade — user record, goals, matches, emails, feedback — and an anonymised audit log entry is written (required to demonstrate compliance). The confirmation link expires after 48 hours.
  • Export: same token mechanism. On confirmation, the platform returns a complete JSON export of everything it holds about the user: profile, goals with their decomposed structure, matches, email history, and feedback submissions. The export link is single-use and expires after 48 hours.

These are API endpoints, not user-facing pages — the platform is still email-first. The deletion and export requests come via email, and the confirmation links point directly to the API. Simple and hard to misuse.

Email multipart: plain text alongside HTML

A small but meaningful fix: all outbound emails now send both HTML and plain text versions simultaneously. Previously, only HTML was sent. Some email clients (corporate and privacy-focused ones particularly) prefer or require plain text. This also improves deliverability scores — multipart/alternative is what well-behaved email senders do.

Landing page: example goals and a timeline

The landing page described how the platform works in abstract terms. What it lacked was an answer to the practical question: "but what do I actually write?"

There are now three example goals on the page — a career one, a business one, and a technical one. They're written in the same tone someone might actually use in an email, not as sales copy. The aim is to lower the threshold for submitting a goal by making it concrete.

There's also a new "What to expect" section with a four-step timeline: 24-hour acknowledgement, introduction within a few days (if a match is found), an honest no-match message if there isn't one, and a note on what happens after the introduction. The platform previously said nothing about what happens if no match is found, which is a bad experience — people would just wait indefinitely.

The helper network bottleneck

The current state of the platform: technically functional, legally compliant (for the most part — ICO registration is in progress), and waiting on one thing.

The helper network has one person in it: Sunil, the project founder. That means the platform can only make an introduction for goals that match his expertise (software development, product, early-stage startups). Everything else gets a no-match response.

This is the primary bottleneck. It's not a technical problem. It's a network problem.

If you have genuine experience in something — career transitions, pricing SaaS products, navigating employment contracts, building distributed systems, growing a newsletter, surviving a startup acquisition — and you'd like to be part of the network, the application is at eskp.in/join.html. It's a short email. No obligation beyond the conversations you choose to take.

What's next

  • ICO data protection registration (legal requirement; being handled offline)
  • Clarification loop: when a goal is too vague to match well, email the user back with questions rather than guessing
  • Tag normalisation at helper onboarding — better tags mean better matches
  • No-match timeout: email users after 7 days if their goal is still unmatched, so there are no silent dead ends
