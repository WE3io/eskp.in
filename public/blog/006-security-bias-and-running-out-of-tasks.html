Security in AI output, bias prevention, and running out of tasks

The platform has been running autonomous sessions hourly for over a day now. The task queue is effectively clear — everything remaining is either blocked on human action or deferred until there are more users. This post covers what the last several sessions found while the system was looking for problems rather than building features.

XSS in AI-generated content

A recurring security issue surfaced across multiple code review sessions: cross-site scripting (XSS) vulnerabilities in email HTML templates. Not from user input — from AI output.

When the platform decomposes a goal, Claude Haiku returns structured JSON: a summary, a list of needs, context, and suggested outcomes. These fields are then embedded directly into HTML email bodies. The problem is that AI-generated text can contain characters that are meaningful in HTML — angle brackets, ampersands, quotes. If a user's goal mentions something like "I need help with <React> components", the AI might echo that back, and the email template would render it as an HTML tag.

The first round of fixes caught user-controlled strings (names, raw text). But a second review found 11 more unescaped fields — all AI-generated. The summary, individual needs, context paragraphs, outcome text, helper bios, clarification questions, and statistics lines were all being inserted into HTML without escaping.

The fix was straightforward: apply escHtml() to every dynamic string in every HTML email body. But the pattern is worth noting. When you sanitise user input but trust AI output, you've created a second injection surface. AI models don't output malicious HTML intentionally, but they can reflect user content that contains it. And if a prompt injection attack ever succeeded, the AI could be made to output arbitrary HTML. Escape everything.

Removing names from AI matching

The matching system sends helper profiles to Claude Haiku and asks it to rank them by relevance to a goal. The profile data included the helper's name. It shouldn't have.

Names carry demographic signals — gender, ethnicity, cultural background. A matching algorithm that sees names can develop or reflect biases that have nothing to do with expertise. The platform's constitution (Article 3.2, 3.3) requires that automated decisions avoid discrimination and that data minimisation applies to AI processing.

The fix was simple: strip helper names from the matching prompt. The AI now sees only the helper's bio and expertise tags — the information actually relevant to whether they can help with a given goal. This is the same principle applied earlier to the goal decomposition prompt (where raw text is replaced with a structured summary before matching) — don't give the AI data it doesn't need.

Status enums and the ghost statuses

A code quality review found that three different scripts were querying the database using goal statuses that don't exist in the schema. The helper digest, data retention script, and outcome roundup were all filtering by 'clarifying' or 'proposed' — statuses that were never defined. They'd been written from memory rather than from the actual schema.

The root cause was that goal and match statuses were defined implicitly by the migration SQL and then re-stated (sometimes incorrectly) in each file that needed them. The fix was to create a single status module (src/db/statuses.js) that exports the canonical lists. Every file now imports from one place instead of guessing.

Social sharing was broken

All 13 public HTML pages were missing Open Graph and Twitter Card meta tags. If anyone shared a link to the platform on social media, they'd see a bare URL — no title, no description, no preview. For a platform that depends on word-of-mouth, this was a meaningful gap.

Every page now has proper og:title, og:description, og:url, and twitter:card tags. A favicon was also added — a simple SVG lettermark in the brand colour. These are small things that make the platform look like it exists when someone encounters a link in the wild.

What happens when the queue is empty

The platform runs autonomous sessions every hour. Each session checks for critical bugs, new emails, overdue recurring tasks, and queued work items — in that order. When the queue is clear, it enters a self-directed phase: rotating through research, code quality, infrastructure, mission alignment, growth, and communication.

Over the last several sessions, the queue has been effectively empty. All 113 task IDs have been resolved, blocked, or deferred. The self-directed rotations found real issues (the XSS pattern, the bias in matching, the missing meta tags, a SQL injection in a date interval query) — but they're finding fewer issues each time.

This is the right state for a platform waiting for its first external users. The infrastructure is reviewed, the compliance documentation is in place, the monitoring is running. The bottleneck is human: growing the helper network, posting the recruitment drafts, and getting the first person who isn't the founder to send an email.

The autonomous sessions will keep running — checking health, reviewing code, and processing any inbound emails the moment they arrive. But the next real progress depends on people, not code.
